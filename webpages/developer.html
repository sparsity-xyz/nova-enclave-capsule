Nova Enclave Capsule

Focus on business logic. Forget the Enclave Ops.

Developing for AWS Nitro Enclaves used to mean writing complex infrastructure code: managing raw VSOCK connections, syncing system clocks with obscure C routines, compiling custom FUSE drivers, and manually proxying egress traffic through the host.

Nova Enclave Capsule is different. We provide a high-level runtime surface that completely abstracts the underlying complexity so you can focus entirely on your application.

The Old Way

  • Handling raw VSOCK & CID routing
  • Building custom proxy layers (squid, socat)
  • Complex AWS NSM SDK integration in C/Rust
  • Fragile time-sync & log plumbing

The Capsule Way

  • ✓ Standard Docker: Bring any runtime (Node.js, Go, Python)
  • ✓ Zero-config Network: Transparent ingress/egress proxy
  • ✓ High-level API: Simple localhost HTTP calls for attestation
  • ✓ Plug & Play: Mount host folders, log to console, automatic clock sync

Architecture & Main Modules

Capsule is designed to wrap your standard Docker image and seamlessly transition it to a secure Nitro Enclave.

Build Path

Capsule CLI

The developer tool. It reads your capsule.yaml manifest, injects the Capsule Runtime into your Docker image, converts it into an EIF (Enclave Image Format), and packages it into the Capsule Shell image.

Host Side

Capsule Shell

The host-side container entrypoint. It wraps nitro-cli to launch the enclave and provides host-side proxies for network egress, filesystem loopback, clock-sync serving, and logs streaming.

Enclave Side

Capsule Runtime

The PID 1 supervisor inside the enclave. It bootstraps the environment, starts proxies, serves the Capsule API locally, mounts host-backed directories, and finally launches your application process.

Runtime & Core Features

What the Capsule Runtime provides automatically to your application running inside the enclave.

🌐

Zero-config Networking

Ingress: Traffic to your enclave port is safely proxied to your app.
Egress: Capsule provides a local HTTP proxy with strict domain whitelisting. Environment variables (http_proxy) are injected into your app automatically.

📁

Host-backed Mounts

Durable or transient file storage. We create a transparent FUSE mount in the enclave that proxies file operations to a loopback disk on the parent instance. Your app uses standard fs.readFile.

⏰

Automatic Clock Sync

Keeps the enclave wall clock accurately aligned with the host—critical for checking TLS certificates, JWT expiration, and time-sensitive cryptographic signing.

📜

Supervision & Logs

Capsule Runtime captures standard output/error and streams it back to the host via VSOCK. It monitors your app's exit status and safely terminates the enclave on exit.

capsule.yaml Overview
version: "v1"
name: "my-secure-app"

ingress:
  - listen_port: 8080        # App listens on 127.0.0.1:8080

egress:
  proxy_port: 10000          # Local proxy port
  allow: ["api.openai.com"]  # Whitelisted destinations

storage:
  mounts:
    - name: "appdata"
      mount_path: "/mnt/appdata"
      required: true
      size_mb: 1024

The Capsule API Suite

A unified localhost HTTP surface. No C/Rust SDKs required. Available at http://127.0.0.1:18000/v1/...

POST /v1/attestation

Generate a cryptographically sound AWS NSM attestation document proving your enclave's identity. Embeds custom JSON user data.

POST /v1/eth/sign & sign-tx

Sign EIP-191 messages or EIP-1559 transactions directly with the enclave's unique hardware-bound Ethereum secp256k1 identity.

GET /v1/random

Fetch 32 bytes of secure entropy backed by the Nitro Secure Module hardware RNG.

POST /v1/encryption/encrypt & decrypt

End-to-End encryption using P-384 ECDH key agreement + AES-256-GCM, ensuring data cannot be intercepted by the host.

Storage & KMS Integration

v1/s3/*

Automated S3 Storage

Instead of wrestling with AWS SDKs and IMDS credentials inside the enclave, use the S3 Capsule API. It proxies S3 requests to a dedicated bucket, enforces key isolation (app-specific prefixes), and can automatically apply KMS-derived encryption to all objects implicitly.

v1/kms/*

Nova KMS

Connect securely to a Nova Key Management Service registry via the Helios RPC module. Use it to derive deterministic, app-bound encryption keys or to securely store persistent Key-Value data in a decentralized KMS environment.

App Wallets

Decentralized Wallets

Capsule allows enclaves sharing the same kms_app_id to cooperatively request an "App Wallet" identity from Nova KMS. You can sign transactions collectively across enclave replicas with no single point of compromise.

Heads Up: To use S3, your egress allow-list MUST include 169.254.169.254 to securely fetch temporary IAM credentials via IMDS through the host.

Getting Started

Developer Guide & Examples

Start fast by utilizing the official templates and example repositories.

Step 1

Bootstrap with Template

Clone the nova-app-template repository. It provides a pre-configured Node.js Express server, Dockerfile, and capsule.yaml. You can write your logic right away without setting up the scaffolding.

Step 2

Explore Code Examples

Check out the nova-app-examples repository. It contains real-world implementations for:

  • AI Prompt Protection: Securely fetching API responses while hiding the prompt details.
  • Encrypted Transport: Generating KMS-secured E2E tunnels to clients.
  • Wallet Orchestration: Signing transactions safely.
Step 3

Mock for Local Development

You don't need a heavy AWS Nitro EC2 instance just to code! Run our Capsule API Mock Server locally to simulate attestation documents, random bytes, encryption, and KMS responses on your MacBook.

Step 4

Build and Deploy

Once you are ready for a real enclave, install capsule-cli and run:

capsule-cli build -f capsule.yaml
capsule-cli run --publish 8080:8080
